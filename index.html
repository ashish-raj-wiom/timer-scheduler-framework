Timer/scheduler framework

Breaking one blind timeout into per-stage schedulers that actually know the ground truth of where a task is stuck.

Stages assigned — several schedulers still proposed, not yet built

The problem

Installation covers a lot of ground before it's done: wiring, config, speed test, fee collection, device linkage, account creation. If the job is pulled away from the field partner mid-sequence, that investment is void and the partner is left worse off for something outside their control.

That's the old timeout. It doesn't care about ground reality — it just says if the job isn't active within the deadline, retrigger. One flat clock, blind to which stage the task is actually sitting in.

That blindness causes a second, sharper problem: during a customer-awaiting stage, it's the customer's move, not the partner's — but the old timeout keeps ticking against the partner anyway. The partner did nothing wrong and can still lose the job.

The fix, in principle

Break the one blind timer into per-stage schedulers, each carrying the action that actually fits that stage's ground truth — instead of one clock that treats every stage the same.

base principle
No RT once both sides have invested.

Once the customer has paid the security deposit and the field partner has completed ID verification, RT (reassignment) comes off the table for every later stage. Taking the job away at that point voids real work already done by both sides, leaving the partner with nothing to show for the effort already put in. "Arrived at site" is the last stage where RT is still allowed — everything from fee payment onward is nudge-only.

The model

Every scheduler is defined by five fields. Multiple schedulers can target the same trigger state — there's no need to chain them into one row; two independent schedulers on "Arrived at site" (one RT, one Nudge) is the normal shape, not a special case.

01
Service
Owning service that runs the check
02
Trigger State
The state that arms the scheduler — the clock starts counting from the moment the task enters this state
03
ETA
Duration counted from Trigger State entry
04
Action States
The states in which the action actually fires. At ETA, if the task is still in one of these, perform the Action — otherwise ignore. A singleton set behaves like a same-state check; a wider set behaves like a gate-check. Size it to fit the Action.
05
Action
What the scheduler actually does when it fires — one of three types, below

Action types

RT

Look for another available partner / reassign the job. Only valid before both sides have invested.

Nudge

Notify the partner to take an action or ask for help. No state change.

Channel (in-app / push / SMS / IVR) is a separate decision, made per-scheduler later.
System Intervention

Trigger an automated operational action outside the core state machine — targets whoever can actually unblock the step, which might be the customer, not the partner.

Not live yet — e.g. a stuck customer-confirmation step → automated call to the customer

There could be more action types in the future — this isn't a closed list.

how to size action states

Nudge / System Intervention → singleton set (just the Trigger State). The message has to match the exact state the task is stuck in — a wider set would blur which one actually applies.
RT → span from the Trigger State forward through every state up to (not including) the point RT stops being legal, per the base principle. RT doesn't care which of those states the task is stuck in, only that it hasn't crossed the investment line yet.

RT implementation checklist

RT isn't one action — it's a sequence. It's already implemented today for the existing acceptance-timeout and installation-retry paths, and both do the same seven things. A new RT scheduler doesn't need new machinery, just a correctly-scoped trigger into this existing pipeline.

  1. Cancel the current task

    The candidate/allocation moves to a terminal-for-this-attempt state.

  2. Free the allocation

    State goes back to unassigned, partner cleared.

  3. Penalize the timed-out partner, two ways

    A cooldown (won't be re-offered this same job immediately) and an assignment-count increment (ping-pong guard, so it can't be re-routed back to them either).

  4. Emit a quality/enforcement signal

    This partner's timeout becomes a scored event on their record. RT is never silent to the partner's standing.

  5. Cancel any lingering timer

    Tied to the old assignment, before a new one is armed.

  6. Re-invoke the routing engine

    Find a new eligible partner, respecting the cooldown just applied.

  7. Two explicit outcomes, both handled
    • Success — new partner found: a fresh RT watchdog is armed on the new assignment. RT is not one-shot; it re-arms itself every time it succeeds.
    • Failure — no eligible partner right now: retry if under the retry cap, otherwise escalate to a routing-failure state that needs manual attention.

Architecture note: this is an async, idempotent, guarded round-trip between services, not one atomic function call. Every step re-checks the entity is still in the expected state before acting (a partner that acted in the gap between timer-arm and timer-fire is a graceful no-op), and there's a backstop sweep in case the round-trip itself is lost.

What this means for the new Arrived-at-Site RT scheduler: the reroute pipeline (cooldown → quality signal → re-route → re-arm) already exists — it's what the installation-retry path already calls once triggered. What's missing isn't machinery, it's a correctly-scoped trigger: today only the old, mis-anchored timeout can kick that pipeline off, and it fires from the wrong point in the journey. The new scheduler's job is to trigger the same existing pipeline from the right point, not build a second one.

deliberate no-RT states
5 of 7 stages: Awaiting Customer Confirmation, Awaiting Partner Assignment, Partner Assigned, Fee Paid, Awaiting Confirmation Code.

RT stays allowed only at Awaiting Slot Proposal and Arrived at Site. The base principle (investment lock-in) only accounts for the last two of those five — Fee Paid and Awaiting Confirmation Code, both after fee payment. The other three are No-RT by separate choice, before any investment has been made — worth writing down the reason if there is one beyond preferring Nudge/System Intervention over churn this early.
